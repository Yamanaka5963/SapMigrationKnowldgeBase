# 18 - テスト戦略 & カットオーバー計画

## テスト戦略

### テストレベル

| レベル | 説明 | 担当 |
|---|---|---|
| **単体テスト** | 個別の設定項目、カスタムプログラム | コンサルタント / 開発者 |
| **統合テスト** | モジュール横断のエンドツーエンドプロセスフロー | コンサルタント + キーユーザー |
| **UAT（ユーザー受入テスト）** | プロセスとデータのビジネス検証 | クライアント業務ユーザー |
| **パフォーマンス / 負荷テスト** | 想定負荷下でのシステムレスポンス確認 | Basis + 技術チーム |
| **リグレッションテスト** | 変換後に既存機能が壊れていないことの確認 | コンサルタント + キーユーザー |
| **カットオーバーリハーサル（モックカットオーバー）** | QAS環境でのカットオーバー手順の完全な予行演習 | 全トラック |

### UAT受入基準（Go/No-Go）
本番稼働前に以下を確認：
- [ ] 合意したUATテストケースの合格率（クライアントと閾値を定義すること、例：95%以上）
- [ ] 未解決のクリティカル（優先度1）課題なし
- [ ] 定義したユーザー群のエンドユーザートレーニング完了
- [ ] マスターデータと未処理トランザクションデータの検証・承認完了
- [ ] ユーザーと権限管理が稼働可能な状態であることを確認
- [ ] 全インターフェースのテスト完了・稼働確認済み
- [ ] インフラとシステム統合チャネルの準備完了

## カットオーバー計画

### 基本原則
- カットオーバー計画は本番稼働の**数ヶ月前**に準備しておくこと — 数週間前では遅い
- 全トラックを特定すること：インフラ、展開、テクニカル（Basis）、ファンクショナル、開発、権限、データ移行
- カットオーバー計画の各タスクには必ず：担当者、所要時間、依存関係、コンティンジェンシーを設定

### カットオーバー計画の構成

| トラック | 主なタスク |
|---|---|
| **テクニカル（Basis）** | システムリフレッシュ、トランスポートインポート、カーネル更新、変換後作業 |
| **データ移行** | レガシーからの最終データ抽出、本番へのロード（マスターデータ → 未処理明細 → 残高） |
| **ファンクショナル** | 設定トランスポート、インポート後の手動設定、モジュール別スモークテスト |
| **権限** | ロール割り当て、ユーザー作成、権限テスト |
| **インターフェース** | 全インバウンド/アウトバウンドインターフェースの有効化とテスト |
| **カットオーバー承認** | クライアントプロジェクトスポンサーとのGo/No-Goミーティング |

### モックカットオーバー
- QASで**最低2回**のモックカットオーバーを実施 — 本番と同じ厳密さで
- 1回目：計画の不備・タイミング問題・漏れステップの洗い出し
- 2回目：修正内容の検証、タイミングと担当の最終確定
- 実際の所要時間を記録し、本番ダウンタイムウィンドウの根拠とする

### 本番稼働当日のシーケンス
1. レガシーシステムの凍結宣言（読み取り専用化）
2. レガシーシステムからの最終データ抽出
3. 本番へのトランスポートインポート
4. データ移行ロード（マスターデータ → 未処理明細 → 残高）
5. インターフェースの有効化
6. 権限割り当て
7. モジュール別スモークテスト
8. クライアントスポンサーとのGo/No-Go判定
9. ユーザーへのシステム開放
10. ハイパーケア開始 → [19 - ハイパーケア・引き渡し](19jp-hypercare-handoff.md)を参照

### ロールバック計画
本番稼働前に必ずロールバックのトリガーと手順を定義しておくこと：
- ロールバックが可能な最終タイミングはいつか？
- ロールバック判断の権限を持つ人物は誰か？
- レガシーシステムを稼働状態に戻すために必要なステップは何か？
