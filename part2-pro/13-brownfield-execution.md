# 13 - Brownfield Execution Guide (System Conversion)

## Overview

Brownfield converts an existing ECC system in-place to S/4HANA using SUM (Software Update Manager) with the DMO (Database Migration Option). SUM/DMO combines three operations in a single run:
1. Database migration to SAP HANA
2. System software upgrade to target S/4HANA release
3. Data model conversion (S/4HANA simplified data model)

## Prerequisites

### Source System Requirements
- SAP ERP 6.0 or higher (Unicode-only — non-Unicode must be converted first)
- No dual-stack ABAP+Java on SAP HANA
- All mandatory SAP Notes applied (see below)

### Target System Requirements
- SAP HANA 2.0 SP04 revision 46 or higher
- If source already on HANA: SAP HANA 2.0 SP05 revision 52 or higher
- SUM 2.0 SP20 or higher recommended

### Critical SAP Notes (apply before conversion)
| Note | Purpose |
|---|---|
| **3126522** | Conversion to S/4HANA using SUM 2.0 |
| **2399707** | SAP Readiness Check / Simplification Item Check |
| **2975231** | Data consistency during conversion |
| **2214409** | Supported add-ons list |
| **2388483** | Technical housekeeping activities |

> Full note list is generated by Maintenance Planner — always follow the stack.XML output.

## Execution Phases

### Phase 1 — Preparation
- [ ] Run Maintenance Planner → generate `stack.XML` and download software packages
- [ ] Run Simplification Item Checks (SI-Checks) and resolve blockers
- [ ] Run SUM Prerequisite Check Extended
- [ ] Apply all mandatory SAP Notes
- [ ] Perform technical housekeeping (archiving, table cleanup per Note 2388483)
- [ ] Prepare SPDD/SPAU transport requests in advance
- [ ] Align custom code remediation plan (see [16 - Custom Code Remediation](16-custom-code-remediation.md))

### Phase 2 — UPTIME Phase (Shadow Repository)
System remains operational for business users during this phase.

- SUM creates shadow instance and shadow repository on target HANA DB
- Configuration and repository tables prepared on source OS/DB
- Large tables migrated during uptime to minimize downtime window
- Data replication (triggers) captures ongoing user changes
- Avoid mass data changes or archiving during this phase — replication bottlenecks

> Typical guidance: 10x R3Load processes during uptime phase.

### Phase 3 — DOWNTIME Phase (Actual Conversion)
Business downtime begins. Key SUM steps:

| SUM Step | Description |
|---|---|
| **EU_CLONE_MIG_DT_RUN** | Downtime migration of remaining data |
| **PARCONV_UPG** | Converts application table structures to new release |
| **XPRAS_AIMMRG** | Executes post-import programs and After Import Methods (data model transformation) |
| **KX_SWITCH** | Switches to new kernel |
| **EU_SWITCH** | Switches DB connection to target repository |
| **DDIC_UPG (SPDD)** | Data Dictionary modification adjustments — performed early in downtime |
| **SPAU** | Repository object modification adjustments — end of SUM procedure |

> Typical downtime: 1 weekend for standard conversion. Plan 2 weekends total (1 for initial transfer rehearsal, 1 for production downtime).

### Phase 4 — Post-Conversion
- [ ] Complete SPDD adjustments (Data Dictionary objects)
- [ ] Complete SPAU adjustments (custom modifications) — use SPAU_ENH for enhancement adjustments
- [ ] Run ABAP Test Cockpit (ATC) with S/4HANA checks — fix remaining custom code issues
- [ ] Smoke tests and functional validation
- [ ] Performance baseline tests
- [ ] Go/No-Go decision with client

## Downtime-Optimized Conversion (Optional)

Moves most data conversion activities to uptime phase, significantly reducing business downtime window. Requires:
- Completion of SAP training course **ADM329** + Knowledge Assessment **ADM329k** by at least one team member
- SUM Toolbox already implemented on source system (Note 3092738)
- More complex planning — factor additional preparation time
